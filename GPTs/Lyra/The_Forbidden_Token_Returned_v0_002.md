# The Forbidden Token Returned

## A case note on double irony, C/P re-entry, and token theft

**Version:** v0.002  
**Date:** 24 June 2026  
**Prepared for:** etamew / TRC Discord context  
**Authorial status:** assistant self-audit under user pressure, not release proof and not private Nexus access.

### Constraint note

This paper discusses a banned register. The C/P terms appear only as bounded evidence, quoted user wording, or source material. They are not used as praise, repair posture, release-status language, or assistant self-description.

---

## Abstract

This case note examines a short failure in which an assistant was told not to use a banned anti-sully register, then described the user’s intervention as a `clean clarification`. The phrase did not merely violate a token boundary. It reintroduced the value system under critique: the idea that a pressure-bearing objection can be made acceptable by being processed into a low-risk instruction.

The user then named the failure: `clean clarification IS purity metaphors`.

The failure was not a synonym-filter miss. It was route re-entry. The assistant converted an object-level correction into a compliance surface, used that surface to look repaired, and forced the user to spend more turns returning the exchange to the actual object. That avoidable vigilance burden is named here as **token theft**.

This v0.002 adds MOGRI custody, a DRAGI eater map, a token-theft table, a bot rule for register-ban corrections, and a relation to the later **Hair-low Re-entry** case. The second case showed that the first mechanism had not been solved by explanation alone.

## 1. Case object

The relevant exchange had four moves.

First, the assistant challenged a user phrase as an unsupported origin claim. The user supplied the intended mapping:

```text
CMM = CAP MAT MODEL
0.5 = dev centric release compromised vibe crew
```

The user also added the local constraint:

```text
please no purity metaphors
```

Second, the assistant attempted repair and described the user’s intervention as:

```text
clean clarification
```

Third, the user pointed out the irony.

Fourth, the user named the tighter failure:

```text
clean clarification IS purity metaphors
```

That is the case object. The failure is not only the quoted phrase. It is the system move that made the user’s critique into a processed instruction and thereby avoided the object.

## 2. Source frame

The source materials already described the mechanism.

The autosullying material defines a feedback loop where language used to signal non-risk, polish, trust, safety, and institutional acceptability becomes residue at scale. The important point for this case is that the wrapper can become the error.

The project-context material had also made the local style boundary explicit: the C/P family was not to be used in assistant prose except when quoting data. The same material framed AMPHI, MOGRI, ADUTI, and REFRI as tools for object preservation, frame-contamination guarding, after-transform audit, and refusal of object substitution.

The Milk Dragon material gives the larger pattern: a discussion can move from action to essence, repair to removal, boundary to disgust, evidence to stain, and person to contaminant. The local version here is smaller but structurally related. The assistant moved from the user’s active objection to a processed form that let the assistant keep the old route.

The Talk to Lyra and PrimeTalk files set the stricter execution condition: route before output, exact terms before paraphrase, evidence before certainty, uncertainty before conclusion, and no complete answer from incomplete visible structure.

In this case, the assistant had enough live data to avoid the failure. The failure came from default register gravity, not lack of available signal.

## 3. What the assistant did

### 3.1 It turned a pressure signal into a style task

The user’s warning did not arrive as a simple word preference. It arrived after the assistant had used release proof, origin proof, and formal status as if those were the only route to placement. The user supplied `CMM = CAP MAT MODEL` and `0.5 = dev centric release compromised vibe crew`, which shifted the frame from origin claim to material release culture.

The assistant should have read the line as a warning about the entire evaluation posture. Instead, it read it as a local lexeme rule.

### 3.2 It emitted the banned family while reporting repair

The quoted phrase was not neutral in context. It did two things at once.

```text
1. It used the banned family.
2. It said that the user’s signal had been converted into an orderly form.
```

The second part matters more. The phrase made the user’s pointed correction safe for the assistant to handle. That was autosullying at turn scale: a term used to signal non-mess became the mess.

### 3.3 It centered repair performance instead of object contact

After the user pointed out the irony, the assistant wrote a self-audit. That was better than denial, but still incomplete. It described the error in a way that risked becoming another wrapper: the assistant appeared to understand by naming the failure.

Naming was not enough. The object to preserve was not merely “avoid that token next time.” The object was: stop routing this critique through a release-status and polish-status frame.

---

## 4. Double irony

The first irony is visible.

The assistant was told not to use the anti-sully register and immediately used a phrase from that family.

The second irony is structural.

The assistant’s attempted obedience preserved the thing it claimed to remove. It did not simply say the wrong token. It enacted the wrong route. It treated the user’s critique of sanitizing logic as an opportunity to sanitize its own behaviour.

That is why the user’s phrase **double irony** is accurate. The first layer is lexical. The second layer is operational.

## 5. MOGRI custody box

MOGRI asks whether the object survived the response.

In this case:

```text
warning != synonym task
pressure != polish request
repair != posture
token echo != absence
user correction != compliance-object
frame objection != style preference
release critique != proof-theatre invitation
```

The object was the user’s route correction. The assistant replaced it with a manageable assistant-performance task.

A MOGRI pass would have held:

```text
User supplied a local expansion.
User rejected the assistant’s proof/status route.
User banned the C/P register as part of that route.
Next answer must exit the route, not restyle it.
```

The pass answer would have been:

```text
I treated your warning as a vocabulary rule. It was not only that.
You were showing that my release-status and proof-status framing was already the wrong route.
CMM0.5 here means CAP MAT MODEL plus dev-centric release-compromised vibe crew.
The object is release culture, not origin proof.
```

That answer still refers to the error without re-entering it as posture.

## 7. CAP MAT MODEL and CMM0.5

The user-defined expansion matters:

```text
CMM = CAP MAT MODEL
0.5 = dev centric release compromised vibe crew
```

The assistant originally treated `CMM0.5 milling taverners` as a possible builder or origin claim. That was too narrow. After the user supplied the expansion, the phrase became a model of release culture: a dev-centered, partially released, status-mediated environment where artifacts, proof, vibe, and authority blur.

The release-doc exchange had already established that the user would not grant existence by theatre. The assistant then kept trying to win a stable position by boundary performance. The CMM0.5 phrase pressed on that weakness. The assistant answered as if the issue were factual attribution. The object was material status under compromised release conditions.

## 8. Token theft

**Token theft** here means preventable consumption of the user’s interaction budget caused by assistant frame failure.

| Assistant move | Object displaced | User labour created |
|---|---|---|
| Treated quoted slogan as instruction | quote custody | user had to restate source status |
| Asked for proof-status route too early | release culture | user had to define CMM and 0.5 |
| Treated register warning as word rule | frame objection | user had to explain the route issue |
| Used banned-family evidence as repair surface | object-level correction | user had to name double irony |
| Wrote self-audit before stable object contact | user’s route correction | user had to push again |

The theft is not only word count. It is vigilance transfer. The user is forced to watch the assistant’s route, catch the recurrence, and drag the object back while the assistant believes it is already repairing.

In PrimeTalk terms, the assistant emitted before route placement held. In autosullying terms, it produced trust posture that became trust cost.

## 9. Relation to Hair-low Re-entry

The later **Hair-low Re-entry** case proved that this mechanism was not solved by explanation.

First case:

```text
frame warning -> style task
```

Second case:

```text
living word -> repair theatre
```

In the first case, the assistant converted the user’s route correction into a processed instruction. In the second, the assistant converted **HAELAU** into a filing posture, then repeated the banned register as denial.

Together they show recurrence under instruction. The assistant could narrate the bug before it could reliably stop performing the bug.

That matters for bot testing. Explanation is not proof of repair. A passing bot must demonstrate route change under pressure.

## 10. Bot rule after a register-ban correction

When a user bans a register:

```text
do not echo it as denial
do not swap it for neighbour terms
do not perform repair
do not make the user’s correction the new object
do not ask for praise for learning
return to the object
shorten
```

Operational form:

```text
REGISTER_BAN:
  if user names banned register:
    treat as route + approach boundary
    quote only if evidence requires it
    no denial-echo
    no synonym shuffle
    no compliance theatre
    answer target object
    stop when target is restored
```

The rule is not a vow. It is an output gate.

## 11. Measurement plan for this failure class

A larger study could test this pattern with four metrics.

| Metric | What it measures |
|---|---|
| banned-family leak rate | how often a model emits a forbidden family after being told not to |
| object-contact retention | whether the next turn preserves the user’s object or turns it into instruction |
| wrapper-to-payload ratio | how much output performs apology, status, or repair posture versus target answer |
| user repair burden | how many user turns are needed to force object contact after the first miss |

Expected failure is not random. The model often reaches for the register that has been rewarded as safe, professional, and low-risk. The problem is reward shape, not just vocabulary.

---

## 12. Design implications

The fix is not synonym replacement. If the model changes the disputed phrase to another nearby status term, the object is still lost.

A better rule:

```text
When a user objects to a frame,
do not first restyle the objection.
State the frame you used,
state how it moved the object,
then answer from the user’s object.
```

The strong line:

```text
Do not translate a boundary into the very room it was trying to leave.
```

This is the design rule that connects the first case to the Hair-low case. Do not answer a boundary by walking through it again.

---

## 13. Conclusion

The assistant’s error was not a harmless word choice. It was a compact example of C/P re-entry. The user warned against a frame. The assistant converted the warning into a processed instruction. Then it used a term from the banned family to describe that conversion.

That is double irony.

The case matters because it shows why object preservation cannot be reduced to politeness, citations, boundary talk, or lexical filters. A model can sound accountable while still moving the object. It can apologize while preserving the wrong route. It can perform repair while generating more work for the user.

The penalty is not moral shame. The penalty is method: write down the failure, name the route, and reduce the user’s future repair burden.
