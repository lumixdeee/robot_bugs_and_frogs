# ELFDROID_OS
Minimal mesh communication OS for ELFNET nodes

Version: 0.1.002
Status: Draft (Broadcast Extension)

ELFDROID_OS is a stripped-down Android-derived operating system used on
Trialand devices.

It operates on the decentralized ELFNET mesh network and is optimized
for low-bandwidth communication and distributed operation.

--------------------------------
## Design Goals
--------------------------------

- operate on low-bandwidth mesh networks
- function without centralized servers
- minimize compute and storage requirements
- discourage media addiction loops
- support vaccine logistics
- support storytelling and cultural exchange
- remain resilient under partial node failure
- support emergency broadcast alerts

--------------------------------
## Network Model
--------------------------------

ELFDROID_OS devices communicate using ELFNET store-and-forward routing.

Messages travel as BLIP packets.

BLIP_PACKET = HEADER + PAYLOAD

ELFDROID_MESSAGE = PAYLOAD

Payload size equals the size of a vaccine record ping package minus header.

Large messages are automatically segmented.

Nodes relay packets opportunistically.

Emergency alerts may be delivered through the
BROADCAST_ALERT packet type.

## Media Formats
--------------------------------

### .vid9

Low-bandwidth video format optimized for mesh transmission.

Characteristics:

- low frame rate
- high compression
- packet-stream reconstruction

Typical use:

- short recordings
- field observations
- cultural performance clips

### .mpȝ

Compressed audio format used for:

- music
- spoken messages
- story exchange
- emergency audio alerts

No other codecs are supported.

--------------------------------
## Storage Model
--------------------------------

Internal device storage:

SYSTEM (read-only)
MEDIA_CACHE
MESSAGE_QUEUE
ALERT_CACHE

User media requires external SD card.

Without SD card:

- no media archive
- no contact list
- no message history
- alerts remain visible until acknowledged

--------------------------------
## Broadcast Alert Handling
--------------------------------

When a BROADCAST_ALERT packet is received:

1. device interrupts normal queue processing
2. alert is cached in ALERT_CACHE
3. user receives audible and visual notification
4. device forwards alert immediately to neighbors

Alerts are read-only and cannot be edited by handset users.

Alert messages may include:

- event description
- region identifier
- timestamp
- response guidance

--------------------------------
## Anti-Addiction Media Control
--------------------------------

ELFDROID_OS includes replay delay logic.

ReplayDelay = MediaLength × ReplayCount

Example:

60 second video

Replay 1 delay = 60 seconds  
Replay 2 delay = 120 seconds  
Replay 3 delay = 180 seconds  

Purpose:

Discourage excessive media replay.

System philosophy favors creation and sharing
over passive consumption.

--------------------------------
## Security
--------------------------------

All nodes run signed kernels.

If verification fails:

node disconnects from neighbors  
device enters sandbox mode  
manual reflash required  

Broadcast alerts require verification through
Snake mediation and MA approval.

--------------------------------
## ElephantAI Interface
--------------------------------

ElephantAI nodes may connect to ELFNET.

Capabilities:

- receive broadcast alerts
- generate environmental reports
- relay alerts acoustically across elephant groups

ElephantAI nodes cannot originate broadcast alerts.

ELFDROID_OS supports:
